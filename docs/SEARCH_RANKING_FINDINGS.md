# Search Ranking Findings

## Reported behavior

Query:

```text
github token
```

Observed top result:

```text
Pepper Medical Log Feburary.docx
```

The result was shown with a large number of entity hits and keyword overlap, even though the document is a medical-log form and is not semantically or lexically relevant to a GitHub token search.

## Relevant architecture

The project intentionally keeps search ranking shared between local and server search paths:

- Local search enters through `src/utils/search.js`.
- Signed-in server search enters through `server/search.js`.
- Both use shared ranking behavior in `src/utils/searchCore.js`.
- Query normalization, aliases, facets, intents, and entity boost selection live in `src/utils/queryUnderstanding.js`.
- Search artifacts are generated by `src/utils/searchIndex.js` and entity extraction by `src/utils/entities.js`.

That shared core is the right place for ranking changes. A UI-only fix or a server-only fix would violate DRY and create inconsistent behavior between guest/local search and authenticated/server search.

## Likely root cause

The ranking bug is caused by broad entity boosting and substring matching combining in a way that rewards unrelated structured documents.

For a query like `github token`, query understanding expands and boosts developer-oriented signals:

- `github` expands to related terms such as `gh`, `workflow`, `actions`, and `repo`.
- `token` triggers the `credentials` facet and boosts credential-related entity types.
- Credential searches currently boost `env_var`, `api_key_like`, and `jwt_like` entities.

The medical-log document contains many uppercase form labels and structured field names. Entity extraction can classify these as `env_var`-style entities because they resemble all-caps identifiers. The current ranking then counts boosted entity types as entity hits even when the entity value does not actually match `github`, `token`, or a meaningful expansion.

This lets a form-heavy document accumulate many entity hits by volume alone.

There is also a substring issue:

- `repo` can match inside `republic`.
- `gh` can match inside unrelated words.

That means an address like `e republic rd` can accidentally contribute to keyword or term overlap for a GitHub-related query.

## Why this ranks incorrectly

The current behavior overweights generic structure:

- Many unrelated form labels can become entities.
- Boosted entity types can count as hits without query-specific value matching.
- Short expanded aliases can match inside normal words.
- Entity-hit count can outweigh direct content relevance.

For developer search, this is especially risky because the index intentionally extracts high-signal technical entities. Those entities should improve precision, but only when the entity value is connected to the query.

## Recommended fix

Fix this in `src/utils/searchCore.js`, keeping local and server ranking behavior shared.

Recommended changes:

1. Make term matching token-aware for simple terms.
   - `repo` should not match `republic`.
   - `gh` should not match inside arbitrary words.
   - Multi-word terms can still use phrase matching.

2. Separate direct entity matches from type-only boosts.
   - Direct matches should count when an entity value matches query terms or expansions.
   - Type-only boosts should be limited to high-confidence credential entities such as `api_key_like` and `jwt_like`.
   - `env_var` should only count when its name actually matches the query language.

3. Cap or dampen entity-hit contribution.
   - A document with 26 weak entity hits should not outrank a document with direct `github` and `token` relevance.
   - Entity count should be a signal, not the dominant ranking mechanism.

4. Add regression coverage.
   - A medical/form-style note with many uppercase labels and `republic` in the address should not outrank a GitHub token note for `github token`.
   - Existing developer-corpus expectations should continue to pass.

## SOLID / DRY notes

- Keep ranking policy in the shared search core instead of duplicating local and server scoring rules.
- Keep query-understanding responsibilities separate from scoring responsibilities.
- Avoid document-title or file-type special cases. The issue is generic ranking behavior, not this one `.docx`.
- Add small helper functions for term matching and entity relevance instead of embedding ad hoc checks inside the scoring expression.

## Suggested test case

Create two notes:

```text
GitHub token note
Content: GitHub Personal Access Token, ghp_xxxxxxxxxxxxxxxxxxxxxxxxx
Categories: github, token, credentials
```

```text
Medical form note
Content: medical-log form text with many uppercase labels and an address containing "republic"
Categories: document
```

Expected behavior:

```text
searchNotesWithArtifacts(notes, "github token", artifacts)[0]
```

should return the GitHub token note, not the medical form note.

## Implementation priority

High. This affects trust in global smart search because an obviously unrelated document can become the top result. The fix should be small and centralized, but it should be protected by regression tests because search scoring changes can easily affect existing benchmark queries.
